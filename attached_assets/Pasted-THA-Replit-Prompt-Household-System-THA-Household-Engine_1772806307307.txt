THA Replit Prompt — Household System + THA Household Engine (Planning Only)
Objective

Design and plan a new Household-first architecture for The Healthy Apples (THA) so the product supports shared meal planning, pantry management, basket optimisation, and future personal food diaries without merging user accounts.

This is a planning and architecture task only.
Do not implement yet.

The goal is to produce a clear build plan for introducing:

Households

Household members and roles

Shared planner / pantry / basket

Individual user goals and future food diaries

THA Household Engine v1

dietary conflict filtering

pantry-aware meal scoring

basket optimisation

A phased implementation plan that avoids major refactoring later

Current Product Context

THA is a meal planning and shopping assistant.

The product already conceptually revolves around:

meal planning

baskets

food decisions

nutrition direction

future diary / tracking potential

The product should no longer assume a single isolated user.
Instead, it should assume food decisions happen at the household level, while still preserving individual user needs.

Examples:

a family shares dinners

two partners share a basket and pantry

users may still log individual lunches later

members may have different allergies, goals, dislikes, and nutrition needs

Critical Product Rule

Do not merge accounts.

Users must remain independent accounts.

Instead, create a Household container that users belong to.

Use this principle throughout the plan:

shared data belongs to household_id

personal data belongs to user_id

Strategic Goal

Position THA not as just a meal planner, but as a household food operating system that can eventually:

resolve dietary conflicts

balance nutrition across multiple people

optimise shopping baskets

reduce waste using pantry data

support collaborative planning

support premium household subscriptions later

Architecture Requirements
1. Household Model

Design THA around a Household entity.

The household should act as the shared container for:

planner

pantry

basket

favourite meals

shared settings later

Each household can have multiple members.

Each member remains an independent user account.

Users must be able to:

create a household

join a household

leave a household

have a role within a household

2. Data Ownership Rules

The plan must clearly separate:

Shared household data

These should generally belong to household_id:

meal planner

basket

pantry

household favourite meals

possible later budget settings

shared routine meals

Personal user data

These should generally belong to user_id:

personal goals

allergies

dislikes

dietary rules

personal nutrition targets

personal diary entries

personal out-of-home lunches

personal weight or progress logs if added later

The response must explain why this separation matters.

The response should also recommend whether meals remain globally reusable or can be household-specific.

This should be part of the THA Household Engine planning.

THA Household Engine Requirements

Ask Replit to design a v1 planning model for the THA Household Engine.

This is a decision layer that helps choose and optimise meals.

The engine should be described as stages, not just as a vague concept.

Required Engine Stages
Stage A — Hard-rule conflict filtering

Meals that violate allergies or non-negotiable restrictions are removed.

Examples:

peanut allergy

vegetarian requirement

vegan rule

medically excluded ingredients later

Stage B — Soft preference scoring

Meals are ranked higher or lower based on:

dislikes

cuisine preferences

ease

repetition avoidance later

family suitability later

Stage C — Pantry boost

Meals should score higher if they use ingredients already in the pantry.

Stage D — Nutrition fit

The system should support the idea that one shared meal can generate different per-person instructions or swaps.

Examples:

tofu for vegetarian

lower-carb portion for weight loss

smaller portion for child

Stage E — Basket optimisation

Once meals are selected:

merge overlapping ingredients

reduce duplicate line items

account for pantry stock where possible

Scoring Logic Requirements

Ask Replit to propose a simple meal scoring model such as:

score =
  pantryMatch
  + preferenceFit
  + nutritionFit
  - conflictPenalty
  - dislikePenalty
  - costPenalty
  - repetitionPenalty

The response should explain which parts are suitable for MVP and which parts can come later.

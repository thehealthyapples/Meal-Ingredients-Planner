THA Replit Prompt — Household Intelligence + Personal Food Diary
Objective

Implement the next THA product phase after Household Foundation and Household Management UI.

This phase has two linked goals:

A. Make the household system feel smart and visible

Add Household Basket Intelligence and Planner-to-Basket traceability so users can clearly see:

who added an item

why an item is in the basket

which planner meal caused it

that basket / pantry / planner are shared household systems

B. Add a Personal Food Diary

Create a user-level Food Diary with a layout closely matching the existing planner UI.

The food diary must:

belong to the user, not the household

allow the user to copy from planner for a day

allow the user to override/edit copied meals

support optional daily recording of:

weight

BMI

mood

sleep

include graphs so users can track progress over time

This phase should make THA feel like:

a smart shared household food platform

plus a personal health reflection tool

The repo already has:

household schema foundation

household ownership swap for planner / basket / pantry / freezer

household management UI

household shared data model

existing planner UI and planner data structures

basket/shopping list flows

pantry/freezer flows

This phase should build on that architecture, not replace it.

Phase 3A — Household Basket Intelligence
Goal

Make shared household shopping behaviour visible and understandable.

Users should be able to tell:

this basket is shared

who added an item

whether the item came from a meal / planner / manual addition

whether the item may already exist in pantry later

why duplicate additions were avoided or merged

Basket Intelligence Requirements
1. Add item attribution

For basket items, surface:

addedByUserId

display name of the member who added the item

If an item was generated from a planner meal or ingredient source, also preserve/display:

source type

source meal name if available

source day / planner context if available

Example UI text:

Added by Colin

Needed for Tuesday dinner

From Chicken Stir Fry

Manual household item

2. Improve duplicate handling

When household members add overlapping items:

use existing normalisation/consolidation logic where available

avoid duplicate household basket lines where the same underlying item is being added

preserve attribution as best as possible

If the system consolidates items, do not lose useful context.

Preferred behaviour:

single basket row

source metadata includes multiple contributing reasons if possible

3. Add household context summary to Basket UI

Add a compact summary panel or banner near the top of the basket page.

Example:

Household: The Clapson Family

Members: 3

Shared basket: active

Shared pantry: active

Keep this subtle and visually clean.

4. Add “why is this here?” context

Where practical, each basket item should show one or more of:

added by member

generated from planner

ingredient source / meal source

pantry-aware later if data exists

This is one of the biggest perceived-smartness improvements.

Suggested implementation approach
Storage / data

Use existing shared basket ownership by householdId.

Where needed, expand basket item fetch logic to join:

users

meal/planner source data

ingredient source records

Do not redesign basket ownership.

UI

Update the basket/shopping list page to show:

item attribution

source explanation

household context summary

Keep response shapes stable where possible, but it is acceptable to expand DTOs returned to the basket page.

Acceptance

User A adds a basket item, User B sees it

basket items show who added them where possible

planner-generated basket items show why they exist

duplicate household additions are cleaner and more understandable

household context is visible on the basket page

no regression to existing shopping list functionality

Phase 3B — Planner to Basket Traceability
Goal

Make it obvious how planner meals influence the shopping basket.

Users should be able to understand:

which meals are contributing ingredients

which day/meal slot they came from

why a given ingredient is in the basket

Requirements
1. Meal-to-basket attribution

Where basket items are planner-derived, show:

meal name

planner day

meal slot if available

Example:

Needed for Tuesday dinner

From Veg Chilli

2. Planner UI context

On the planner side, show a small indication when a meal contributes ingredients to the shopping list.

Example:

“Added to household basket”

“3 ingredients linked”

Do this only if it can be added cleanly without clutter.

3. Preserve manual override behaviour

If users manually edit basket items later, do not break the original planner source relationship unless necessary.

If source links must be one-way only, document that clearly in the implementation summary.

Acceptance

at least one clear planner → basket link is visible in UI

basket source explanations are understandable

no current planner behaviour breaks

Phase 3C — Household Dietary Conflict Groundwork
Goal

Lay the groundwork for household-aware food intelligence.

This phase does not need a full recommendation engine yet, but should support future safer meal suggestions.

Requirements
1. Aggregate household dietary context

Use existing user/household data to prepare a basic aggregated household profile where possible:

diets

allergens

restrictions

If this data already exists in user preferences or member data, use it.
Do not overengineer.

2. Expose useful context in planner or meal suggestion flows

Where meal suggestions are already generated, add light groundwork for:

household restriction awareness

future filtering/scoring

This can initially be backend scaffolding plus a light warning or eligibility flag.

Example:

“May conflict with household dietary preferences”

“Suitable for all household members” later

3. Do not block shipping on full intelligence

This is groundwork, not a full recommendation engine.

Acceptance

household dietary context is not lost

future meal filtering/scoring becomes easier

current planner flow remains stable

Phase 4 — Personal Food Diary
Goal

Create a Food Diary that is:

user-specific

visually very similar to the planner

able to copy meals from the planner for a selected day

editable after copying

able to store optional wellbeing and body metrics

able to show progress trends

This is personal, not household-owned.

Food Diary Core Rules
Ownership

Food diary data belongs to userId, never householdId.

Layout

The page should feel very similar to the planner:

date-based rows/cards

Breakfast / Lunch / Dinner / Snacks structure

easy to scan and edit

Copy from planner

Users can choose:

copy today’s planner meals into diary

copy any selected day’s planner meals into diary

Copied items become diary entries that can then be edited independently.

This is crucial:

planner = shared plan

diary = what this user actually ate

So after copying:

user may edit meal names

change quantities

delete entries

add totally different foods

Food Diary Schema / Data Model

Add user-owned diary tables.

Recommended structure:

food_diary_days

Represents one user’s diary day.

Suggested fields:

id

userId

date

optional daily notes

optional daily summary fields

food_diary_entries

Represents individual diary items/meals.

Suggested fields:

id

dayId

userId

mealSlot (breakfast, lunch, dinner, snack)

name

notes optional

sourceType (manual, copied_from_planner)

sourcePlannerEntryId nullable

optional nutrition-related fields if easy to support later

food_diary_metrics

Optional daily health / wellbeing metrics.

Suggested fields:

id

userId

date

weight

bmi

mood

sleepHours

optional sleepQuality

optional energyLevel

optional notes

You may also choose to merge daily metrics into food_diary_days if that fits the repo better.
Use the cleanest implementation for current architecture.

Food Diary Features
1. Planner-like diary layout

Build a diary page that resembles the planner.

Suggested structure:

date selector / week navigation

daily cards or rows

meal slots:

Breakfast

Lunch

Dinner

Snacks

2. Copy from planner

Users can:

copy all meals from planner for the selected date

copy once without duplicating endlessly

then edit diary independently

Important:

copied entries should not stay live-linked to planner forever

they should become editable diary records

keep source metadata for traceability

3. Manual editing

Users can:

add diary items manually

rename copied items

remove copied items

add notes

reflect what they actually ate, not just what was planned

4. Optional metrics

On each diary day, allow optional recording of:

weight

BMI

mood

sleep

Also strongly consider adding:

energy level

daily notes

hunger/cravings marker

adherence / “stuck to plan” toggle

Add these only if they fit cleanly.

These should be optional and unobtrusive.

Phase 5 — Food Diary Progress Graphs
Goal

Add useful visual progress tracking for personal health reflection.

Graphs should be user-level only.

Required graphs
1. Weight graph

Track weight over time.

2. BMI graph

Track BMI over time.

3. Goal progress graph or indicator

If the user has goals set, show progress relative to:

target weight

target BMI

or general direction of change if target exists

If goal data already exists in user preferences, use it.

Recommended additional graphs / insights

If easy and sensible, also include:

average sleep trend

mood trend

weight change over 7 / 30 days

BMI change over 7 / 30 days

consistency / diary completion rate

planner vs diary adherence later

Use judgement. Do not clutter the UI.

Graph implementation rules

Use existing charting approach in the repo if one exists

keep visuals clean and consistent with THA styling

mobile readability matters

avoid overloading the user with too many charts at once

A good first version:

one summary card row

one main graph

one secondary graph

small insight cards underneath

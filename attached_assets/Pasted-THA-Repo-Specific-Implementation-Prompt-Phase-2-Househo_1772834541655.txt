Product rules

These rules are the source of truth for implementation:

1. One active household per user

A user may belong to only one active household at a time.

2. Accounts are never merged

Users remain separate accounts.
Only shared household data changes.

3. Shared vs private data

Shared:

planner

shopping list

shopping extras

pantry

freezer meals

Private:

personal preferences

health data

personal goals

personal history

future diary entries

4. Joining does not auto-merge old household data

If a user joins another household:

do not merge their old solo household planner/basket/pantry/freezer data into the new household

their old household may remain in the DB

their active membership changes to the new household

5. MVP membership roles

Support:

owner

admin

member

For this phase:

owner can rename household

owner can remove members

owner can view/copy invite code

admin/member can view household

join flow assigns new users role member

6. Safety rule for leaving

If a user leaves a household, they must always end up with an active household.
For MVP:

leaving a household auto-creates a new solo household for them

they become owner of that new solo household

7. Owner restriction

If the owner is the only member:

they may leave, but should just remain/convert to their own solo household without data loss

If the owner is in a multi-member household:

for MVP either:

block leaving until ownership transfer exists, or

allow leave only if they are the only member

Use the simpler, safer option:
block owner leave when other active members exist

T203 — Implement household API endpoints

Add routes in server/routes.ts.

Required endpoints
GET /api/household

Returns current user’s active household plus members.

Response should include:

household id

household name

invite code

current user’s role

members list with:

userId

display name / username / email, depending on available fields

role

status

POST /api/household

Creates a new household for the current user only if needed.

Preferred behaviour:

if user already has an active household, do not create a second active one

either return current household or reject with a clear message

Input:

{ "name": "The Smith Household" }
POST /api/household/join

Join household by invite code.

Input:

{ "inviteCode": "ORCHARD8" }

Server behaviour:

validate invite code

find target household

ensure user is not already in that household

ensure user can switch households safely

switch active membership to target household

return joined household summary

Important:

do not merge old household shared data into new household

keep personal user data untouched

POST /api/household/leave

Leaves current household.

Server behaviour:

get current active household

if user is owner and other active members exist, block with clear message

otherwise create new solo household

move user into it as owner

old shared data stays with old household

PATCH /api/household

Rename household.

Input:

{ "name": "The Healthy Apples Home" }

Owner only.

DELETE /api/household/members/:userId

Owner removes another member.

Server behaviour:

verify actor is owner of household

verify target is active member of same household

target cannot be the same as actor through this endpoint

remove target from household

create new solo household for target

return updated member list

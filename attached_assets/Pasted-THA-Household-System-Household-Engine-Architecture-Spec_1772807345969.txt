THA Household System + Household Engine
Architecture Specification — Approved v2

This document defines the household architecture for The Healthy Apples (THA) and the phased introduction of the THA Household Engine.

No implementation should begin until this architecture is accepted.

Section 1 — Executive Recommendation

THA should adopt a household-first architecture by introducing a Household container that owns shared food decisions.

This is implemented as an additive layer, not a rewrite.

The architecture rule is:

Shared decisions → householdId
Personal data → userId

This allows THA to support:

shared meal planning

shared grocery baskets

shared pantry tracking

collaborative planning

individual goals and food diaries

Accounts are never merged.

Users remain independent accounts that belong to a household.

MVP Recommendation

Phase 1 introduces:

households

household members

planner ownership by household

basket ownership by household

pantry ownership by household

The Household Engine will be introduced only after shared ownership is stable.

Section 2 — Domain Model

The system is structured around five primary objects.

User
 └── HouseholdMember
        └── Household
              ├── Planner
              ├── Shopping Basket
              ├── Pantry
              └── Future: Favourite Meals

Meals remain independent.

Meals
 ├── System meals (global)
 └── User meals (personal)

Planner entries reference meals but may later store snapshots for stability.

Core Principles

Household membership rule

A user may belong to one household at a time.

Section 6 — THA Household Engine

The Household Engine selects meals using a filter + scoring pipeline.

Engine rules must be planner-slot aware.

Example:

Dinner slot → strict shared rules
Future personal lunch → personal rules

Stage A — Hard Rule Filtering

Remove meals violating any member restriction.

Examples:

peanut allergy

vegetarian restriction

vegan restriction

Union all restrictions across household.

Stage B — Preference Scoring

Rank remaining meals.

Example scoring:

+2 cuisine match
+1 quick meal
+1 household favourite
-1 dislike match
-2 repetition penalty
Stage C — Pantry Boost

Meals gain score if pantry ingredients exist.

+0.5 per matching ingredient
max +3

This reduces food waste.

Stage D — Nutrition Fit (Phase 2)

Future feature:

Adjust portions or swaps.

Example:

Alice → tofu swap
Bob → reduced carbs
Child → smaller portion
Stage E — Basket Optimisation

After meals selected:

gather all meal ingredients

normalise ingredients

merge duplicates

subtract pantry stock

generate final basket

Basket merge safety rule

Only merge ingredients if:

ingredient normalization confidence high

unit conversion supported

Otherwise keep separate.

Example:

2 onions + 500g onions → uncertain → do not merge
Section 7 — Migration Strategy

Migration uses deterministic user mapping.

Do not join households by name.

Step 1 — Create households

Each existing user gets a default household.

INSERT INTO households (name, inviteCode, createdByUserId)
SELECT
  username || '''s Household',
  substr(md5(random()::text),1,8),
  id
FROM users
Step 2 — Create memberships
INSERT INTO household_members
(householdId, userId, role)

SELECT
  households.id,
  users.id,
  'owner'
FROM users
JOIN households
ON households.createdByUserId = users.id
Step 3 — Add householdId columns

Add to shared tables.

Backfill using household_members.

Step 4 — Keep userId columns

Do not drop immediately.

Run both fields in parallel during transition.

Section 9 — Risks and Mitigation
Risk	Mitigation
migration errors	keep userId columns
incorrect household join	preview household before joining
invite code abuse	add invite table in future
pantry subtraction errors	floor quantities at zero
duplicate basket items	merge using normalizedIngredients
owner deletion	block deletion if members exist
Meal ownership risk

Meals remain user-owned.

If a user leaves the household:

Planner entries referencing their meals remain valid because:

Planner entries may store a snapshot reference.

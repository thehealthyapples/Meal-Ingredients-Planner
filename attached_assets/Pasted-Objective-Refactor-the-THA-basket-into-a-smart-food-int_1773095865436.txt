Objective

Refactor the THA basket into a smart food-intent system. Treat whole foods and packaged foods differently. Add multi-supermarket selection, a global basket tier, a client-side ingredient catalogue, whole-food variant selectors, a score-based matcher + fallback chain, and a per-row Food Confidence Indicator. Keep existing packaged-food flows intact. Implementation is additive: new lib/modules + targeted changes to schema, routes, and shopping-list-page.tsx.

Core product framing

THA optimises food choices and basket intent internally.
Supermarkets fulfil the closest available products based on availability and substitutions.

Do not regress packaged-food behaviour.
Do not do a giant rewrite.
Keep changes modular and config-driven where practical.

Architecture notes

- shopping-list-page.tsx already has:
  - PriceTier
  - TIER_LABELS
  - SUPERMARKET_NAMES
  - per-item selectedTier
  - exportSupermarket (single)
  - globalStore
- Add basket-level state:
  - selectedRetailers: string[] (multi-select, persisted in localStorage)
  - globalBasketTier: PriceTier | "item" (persisted in localStorage)
- Use stable retailer ids in state, not display labels
- Add nullable schema columns to shoppingList:
  - itemType
  - variantSelections
  - attributePreferences
  - confidenceLevel
  - confidenceReason
- Confidence is derived state and should be recomputed client-side when selectors, selected retailers, or effective tier change
- All new logic should live in new client lib files where possible
- Existing PATCH route at api.shoppingList.update.path must accept the new fields
- Shopping-list generation routes should stamp itemType using a simple heuristic
- Packaged-food flows must continue working as they do today

T003: Whole-Food Logic Modules

Files:
- client/src/lib/whole-food-matcher.ts
- client/src/lib/whole-food-fallback.ts
- client/src/lib/food-confidence.ts

Implement whole-food-matcher.ts:
- Define WholeFoodIntent:
  - ingredientName
  - variantSelections
  - attributePreferences
  - tier
  - selectedRetailers
- Export:
  - scoreCandidate(candidate, intent, tier, selectedRetailers): number
  - resolveBestMatch(candidates, intent, tier, selectedRetailers): ProductMatch | null
- Logic:
  - hard filter out candidates not in selectedRetailers
  - score variant match positively
  - score requested attribute match positively
  - score tier alignment positively, especially organic tier
  - penalise price-tier mismatch
- Keep scoring weights as constants at the top of the file for easy tuning

Implement whole-food-fallback.ts:
- Export buildFallbackChain(intent: WholeFoodIntent): string[]
- Example for organic Pink Lady apples:
  - "organic pink lady apples"
  - "organic apples"
  - "pink lady apples"
  - "apples"
- Logic:
  - start with full intent
  - relax attribute
  - relax variant
  - fall back to bare ingredient

Implement food-confidence.ts:
- Define ConfidenceLevel = "exact" | "close" | "substitution"
- Export:
  - calcConfidence(intent, candidates, selectedRetailers): { level, reason }
  - CONFIDENCE_LABELS
- User-facing labels:
  - exact => "Exact match likely"
  - close => "Close match likely"
  - substitution => "Substitution likely"
- Rules:
  - exact: 2+ viable candidates and top match satisfies key variant + requested attribute
  - close: at least 1 viable candidate or broader retailer pool gives a near match
  - substitution: only weak/fallback candidate exists

Acceptance:
- modules compile
- better matches score higher
- confidence results are sensible

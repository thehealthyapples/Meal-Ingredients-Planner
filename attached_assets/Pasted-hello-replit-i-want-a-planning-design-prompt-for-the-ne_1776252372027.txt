---

## CONTEXT

THA already has:

* household-linked user accounts and profiles
* planner as the main planning surface
* cookbook accessible from planner
* recipes/meals chosen in the normal way

i do NOT want a separate meal generator flow.

i want THA to stay planner-first:

* meals are chosen as normal in planner / cookbook
* then AI adapts that chosen meal for the people who are actually eating it

the key idea is:

* one shared household meal
* slight plate-level adaptations per eater
* not separate totally different dinners

examples:

* tuna spaghetti -> vegetarian eater gets a vegetarian swap, keto eater gets keto noodles
* egg salad -> high-protein eater gets extra eggs / beans / tofu
* pescatarian + vegetarian household -> same bowl, different protein on top
* coeliac requirement -> base meal and all adaptations must remain gluten safe

---

## NEW REQUIREMENT

planner needs a temporary per-person override layer.

example:

* Colin default profile = high protein
* Lindsay default profile = vegetarian
* but for this week both want vegan

so THA must support:

1. profile defaults
2. planner-level temporary overrides

priority should be:

1. hard restrictions always win

   * coeliac
   * allergies
   * intolerances
   * medical exclusions

2. planner override wins over default soft preference

   * vegan this week
   * pescatarian this week
   * high protein this week
   * keto this week

3. profile default applies if there is no override

   * vegetarian
   * high protein
   * pescatarian
   * keto
   * etc.

---

## GOAL

design a lean implementation plan for:

A. planner stays the primary meal-selection flow
B. each planner meal knows who is eating it
C. THA can adapt that one chosen meal for those eaters
D. per-person planner overrides can temporarily replace default profile preferences
E. shopping can later account for extra swap ingredients if needed

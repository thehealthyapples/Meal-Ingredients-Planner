THA Repo-Specific Implementation Prompt — Phase 1B Household Ownership Swap
Objective

Implement Phase 1B of the THA Household System.

Phase 1A is already complete. This phase switches shared data ownership from userId to householdId for:

planner

shopping list

shopping list extras

pantry

freezer meals

Rules for this phase:

all reads query by householdId

all writes persist both userId and householdId

solo-household users must see zero behaviour change

no UI changes

no schema changes

Context

Phase 1A already delivered:

households

household_members

household_id on the 5 shared tables

legacy backfill for all existing users

server/lib/household.ts with getHouseholdForUser()

storage helper support in server/storage.ts

This means Phase 1B is an ownership-key swap, not a redesign.

Use the existing repo structure:

server/storage.ts

server/routes.ts

server/lib/household.ts

Do not create parallel systems unless absolutely necessary.

Critical rules

No UI changes

No schema changes

No route path changes

No response shape changes

No behavioural changes for current users

Do not remove userId from any shared table

Preserve userId for attribution

Use getHouseholdForUser(userId) as the source of truth

Scope all update/delete operations to the caller’s household

Leave meal_plans, user_preferences, user_streaks, user_health_trends, product_history, and personal meals untouched

Implementation guidance
Preferred pattern

For shared reads:

const householdId = await getHouseholdForUser(userId);

Then query by householdId.

For shared writes:

const householdId = await getHouseholdForUser(userId);

await db.insert(...).values({
  userId,
  householdId,
  ...
});

For update/delete-by-id:

resolve household

ensure the target row or its parent belongs to that household

only then mutate

Planner-specific guidance

If planner child tables do not contain householdId directly:

do not redesign schema in this phase

enforce household scope through parent planner week ownership

Example:

entry → day → week

verify week.householdId === callerHouseholdId

Keep changes minimal

Prefer:

small ownership swap

storage-driven changes

minimal route edits

no duplicate service layers

no unnecessary abstractions
